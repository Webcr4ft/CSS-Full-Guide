# What Are User Requirements, User Research, and Testing?

Understanding **user requirements**, **user research**, and **user testing** is an essential part of building websites and applications that people enjoy using. Although these three concepts are closely related, they each serve a different purpose during the design and development process.

---

# What Is User Research?

**User research** is the systematic process of studying the people who use your product.

The main goal of user research is to understand:

* What users need.
* How users behave.
* What problems (pain points) they experience.
* What users like and dislike.
* How satisfied users are with your product.

Instead of making assumptions about users, developers collect real data that helps improve the product.

---

# Why Is User Research Important?

User research helps developers:

* Understand their target audience.
* Discover problems users face.
* Improve the user experience.
* Reduce guesswork during development.
* Build products people actually enjoy using.

Without user research, developers may spend months creating features that users never wanted.

---

# Common Types of User Research

There are many different ways to collect information from users.

Some of the most common include:

* Net Promoter Score (NPS)
* Exit Interviews
* User Surveys
* User Feedback
* Analytics

---

# Net Promoter Score (NPS)

One of the most common user research methods is the **Net Promoter Score (NPS).**

NPS measures:

> **How likely a user is to recommend your product to someone else.**

This gives companies an idea of how satisfied users are.

---

# How NPS Works

Users answer a simple question such as:

> **"How likely are you to recommend this product to a friend?"**

They select a score from:

* **0** = Not likely at all
* **10** = Extremely likely

The survey usually appears at important milestones, such as:

* After using the product for **7 days**
* After **30 days**
* After **90 days**

This allows companies to see how user satisfaction changes over time.

---

# Understanding the NPS Scale

The score ranges from **0 to 10.**

Generally:

* **9–10** → Active Promoters
  * Love the product.
  * Recommend it to others.

* **7–8** → Passive Users
  * Satisfied but not enthusiastic.

* **0–6** → Detractors
  * Unhappy users who are unlikely to recommend the product.

The more promoters a product has, the healthier the product usually is.

---

# Exit Interviews

Another popular research method is the **Exit Interview.**

This is a survey shown when users:

* Cancel a subscription
* Delete an account
* Stop using a service

Instead of guessing why users leave, companies simply ask them.

---

# Why Exit Interviews Matter

Exit interviews help developers discover:

* Missing features
* Expensive pricing
* Poor customer support
* Bugs
* Confusing interfaces
* Better competing products

This information can reduce **user churn**.

---

# What Is User Churn?

**User churn** means users stop using your product.

For example:

* Someone cancels Netflix.
* Someone deletes Instagram.
* Someone uninstalls your app.

Understanding why users leave helps companies improve their products.

---

# What Is User Testing?

While user research asks users questions, **user testing** observes users interacting with the product.

Instead of asking:

> "Do you like this?"

Developers watch users actually use it.

This reveals problems users may not even notice themselves.

---

# Goal of User Testing

User testing helps developers discover:

* Confusing layouts
* Broken features
* Navigation issues
* Slow workflows
* Accessibility problems
* Bugs

Watching users interact with a product often reveals issues surveys cannot.

---

# Example of User Testing

A common example is **beta testing** for video games.

Players receive an unfinished version of the game and play normally.

Developers collect information about:

* Bugs
* Crashes
* Gameplay balance
* Performance
* User feedback

After fixing the issues, the final version is released.

---

# A/B Testing

Another common form of user testing is **A/B testing**.

This involves comparing two versions of a feature.

For example:

Version A:

* Blue "Buy Now" button

Version B:

* Green "Buy Now" button

Different users see different versions.

---

# How A/B Testing Works

1. Randomly divide users into groups.
2. Show Version A to one group.
3. Show Version B to another.
4. Measure which version performs better.

Developers often compare:

* Click rates
* Purchases
* Sign-ups
* Time spent on a page
* User engagement

The better-performing version becomes the default.

---

# What Are User Requirements?

**User requirements** describe what an application must do and how it should behave.

Think of them as a checklist developers follow while building software.

They guide the development process from beginning to end.

---

# Where Do User Requirements Come From?

User requirements can be created from several sources, including:

* User research
* User testing
* Industry standards
* Stakeholder requests
* Business goals
* Customer feedback

Requirements often evolve as more information becomes available.

---

# Functional Requirements

A **functional requirement** describes **what the application should do.**

Examples include:

* Users can log in.
* Users can reset passwords.
* Users can upload profile pictures.
* Users can search products.
* Users can make payments.

These describe the application's features.

---

# Non-Functional Requirements

A **non-functional requirement** describes **how the application should behave.**

Examples include:

* Pages should load within two seconds.
* The website should be secure.
* The application should support 10,000 users simultaneously.
* The site should be accessible.
* The system should remain reliable.

These focus on quality rather than specific features.

---

# User Requirements Are Not Permanent

User requirements change over time.

As developers collect more information from:

* User research
* User testing
* Analytics
* Customer feedback

they often update the requirements to better serve users.

A product should evolve alongside its users.

---

# User Research vs User Testing vs User Requirements

| Concept | Purpose |
|---------|----------|
| **User Research** | Understand users, their needs, behaviors, and pain points. |
| **User Testing** | Observe users interacting with the product to identify usability issues. |
| **User Requirements** | Define what the application should do and how it should behave. |

---

# Why Understanding These Concepts Matters

Knowing the difference between user research, user testing, and user requirements helps developers:

* Build better products.
* Solve real user problems.
* Make informed design decisions.
* Improve usability.
* Increase user satisfaction.
* Reduce user churn.
* Continuously improve the application.

Collecting accurate information from real users leads to products that are more useful, more enjoyable, and more successful.
